Augmented ARchitecture is a demo app that showcases how Apple’s ARKit can be used to natural, meaningful educational experiences. The app helps people learn about notable buildings from around the world by interacting with physical models.

After launching the app on an iOS device, students simply point their device’s camera at one of several models. They can then interact with virtual content to learn about each building.

The experience is designed to teach students about the history, purpose and design of each structure. Because the experience is rooted in augmented reality, information and interactions are presented in-situ: details about the structures are highlighted at the corresponding place in each model.

DESIGNING THE EXPERIENCE

I created Augmented ARchitecture to both explore and demonstrate how augmented reality can be used to help students learn about real, physical objects. To that end, my design process centered around a few key priorities:

EXPLORING HOW PRESENCE CAN ADD TO THE LEARNING EXPERIENCE

Too often, augmented reality is used as a mere gimmick. I’ve seen many examples of experiences where virtual objects feel tacked on to an environment in a misguided attempting at feeling cool, novel, or ‘immersive’.

I believe that placing virtual objects in physical space is only valuable insofar as that process serves some user purpose. While creating Augmented Architecture, I set out to examine how the interaction between atoms and pixels could actually help students learn about structures.

In the experience, content is placed in-situ: there are direct and meaningful connections between the physical models in the world and the information and interactions on the screen.

Users launch each experience by looking at specific models. Content is triggered by tapping on specific locations in the models. Users can view photographs of the real buildings from accurate positions in an around the buildings.

MINIMIZE FRICTION

I believe that the most effective user interface is whatever requires the least interaction to achieve a goal. In the context of augmented reality, this means removing contrivances like scanning QR code’s or manual alignment of content.

In Augmented ARChitecture, interface friction is minimized by allowing objects themselves to function as content triggers. From the perspective of users of the app, selecting content is as simple as pointing a camera.

TESTING MODES OF INTERACTION

One of the primary benefits of augmented reality is its ability to make a more natural user interface. But experience still have some interface. One of the goals for this project was to explore the nature of those interactions, and how to best communicate how to use the app.

For many users, augmented reality is an entirely novel paradigm for interaction. There are very few well-established conventions that users can intuit about an experience. So designers must communicate that users can access content using physical objects, how to access that content, and which objects can trigger content.

THAT  PHYSICAL OBJECTS CAN LINK TO CONTENT

For new users, the connection between objects in the physical environment and virtual content may be non-obvious.

The default view of the application looks like something familiar: a phone’s default camera app. Yet it functions differently: instead of capturing and storing images of the world, it acts as an interface to the physical world.

This may seem like a simple concept, but it requires a drastic change in the mental model that people have about phones, cameras, and computing in general. To address this, I created a simple set of graphics that are displayed to the user.

In user testing and observation, I’ve found that this is a lesson that only needs to be learned once. In general, once a person understands that looking at objects works as a link to content, they can use AR applications without further difficulties.

HOW  TO TRIGGER CONTENT

While designing Augmented Architecture, I had intended to allow users to access additional information by tapping on appropriate positions on the device’s screen. But I did not anticipate that I was relying on some non-obvious assumptions in my design.

During (informal) user testing and observation, I noticed that several people interpreted messages like “Tap on an object” to refer to the physical objects themselves, rather than the images of the objects on the screen. They would tap on the model of a building, and be confused or frustrated as to why nothing was happening in the application.

In retrospect, it’s a pretty obvious point of confusion. The entire point of augmented reality is to blur distinctions between the digital and the physical. Yet some interactions must necessarily occur in the context of a device, rather than in the world itself.

I attempted to address this issue by adding more salient visual cues like high-contrast, animated hotspots within the experience, and by adding additional copy to the app’s instruction process.

WHICH  OBJECTS TRIGGER CONTENT

During user observation, I discovered that the most powerful tool for communicating which objects trigger content is the design of physical environment itself. Rather than attempting to explain this idea digitally, potential triggers should be physically, spatially, and conceptually distinct from other objects in the immediate vicinity.

In Augmented ARchitecture, the objects that trigger content are all plastic models, of buildings, that are near one another. Every object in the space that fits any of those criteria can trigger content, and no other object fits any of those criteria.

CREATING THE EXPERIENCE

The code for Augmented Architecture was written in Swift using Xcode. The git repo for the project is available here.

The demo uses standard iOS libraries including ARKit, SceneKit, SpriteKit, and AVFoundation.

3D Assets were created in Blender, then exported as COLLADA (.dae) files. These were converted to SceneKit scene descriptions (.scn) in Xcode.

Simple videos for the project were created using iMovie, then imported to the scene as SpriteKit video nodes.

TARGET OBJECTS

Because minimal UI friction was a priority for this project, I needed to create AR target definition files (.arobject) for each building model. These were created using a slightly-modified version of Apple’s AR Scanner example app.

MODEL OCCLUSION

Maintaining a realistic sense of physicality in virtual objects is one of the most important design considerations for augmented reality experiences. And one of the most important aspects of realism is occlusion: objects in the physical world need to realistically block virtual objects behind them.

Unfortunately, this can be a difficult task; ARKit isn’t yet capable of automatically handling occlusion of objects. So I had to add it to my experience itself.

This required creating 3D models of each of the buildings used in the experience. These models are aligned to their physical counterparts in the ARKit scene, and given special ‘holdout’ materials in SceneKit. This special holdout material prevents the ARViewController’s renderer delegate from showing any content that overlaps with the occlusion models.
